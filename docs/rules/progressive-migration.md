# 渐进迁移开发规范

本文定义项目所有新框架引入、新功能开发、旧功能重构必须遵循的**渐进迁移流程**。

> **核心原则**：禁止大爆炸式重写（Big Bang Rewrite）。任何重构必须以可独立编译、可独立验证的阶段推进，确保主线始终处于可构建状态。

---

## 为什么需要渐进迁移

大爆炸式重写的典型风险：
- 重写期间主线长期不可构建，阻塞其他工作
- 功能回归难以定位（"改之前能用，改完不能用"，但改动跨越数百行）
- 重写中途发现设计问题，已无法回退
- 新旧接口并存时产生混乱依赖

渐进迁移通过**阶段隔离**消除上述风险：每个阶段结束时，代码库处于一个清洁、可验证的状态。

---

## 四阶段流程

### 阶段 1 — 新建（Build Alongside）

**目标**：新模块可独立编译，但不影响任何现有功能。

**操作**：
- 在要重构的模块**同级目录**下新建独立文件夹（如 `xxx-v2/`）
- 在该文件夹内实现新模块的完整代码
- 将新文件夹加入项目 CMakeLists.txt（使用 `zephyr_library_named` 等 Zephyr 原生 API）
- 确保整个项目编译通过
- 新模块的功能**全部禁用**（通过 `CONFIG_RTFRAME_XXX_V2=n` 控制）

**验收**：`make cm7` / `make cm4` 零错误通过。

**commit 格式**：`[FEAT] add <module>-v2 skeleton`

**禁止**：
- 阶段 1 中不允许修改老模块任何代码
- 不允许新模块 include 老模块的内部头文件
- 不允许提交含编译错误的代码

---

### 阶段 2 — 并行（Parallel Enable）

**目标**：新旧模块同时存在，逐步将功能从老模块迁移到新模块，通过开关控制。

**操作**：
- 通过 Kconfig 开关，关闭老模块的**部分**功能，同时开启新模块对应功能
- 两套代码并存于代码库，可随时通过开关切换
- 针对切换后的功能进行对比测试

**开关示例**：
```cmake
zephyr_library_sources_ifdef(CONFIG_RTFRAME_UART_V2 uart_v2.c)
zephyr_library_sources_ifndef(CONFIG_RTFRAME_UART_V2 uart.c)
```

**验收**：
- 新开关 ON：功能正常，编译通过
- 新开关 OFF：等价于阶段 1，与主线无差异

**commit 格式**：`[FEAT] enable <module>-v2 partial`

**禁止**：
- 不允许删除老模块代码（留待阶段 4）
- 不允许两套代码在同一编译单元中同时激活（互斥开关）

---

### 阶段 3 — 验证（Verify）

**目标**：确认新模块功能完整、无回归，形成书面记录。

**操作**：
- 运行所有相关测试（单元测试 / 集成测试 / 烟雾测试）
- 对比新旧模块在相同输入下的输出
- 在对应计划文件（`docs/rules/plans/<topic>.md`）末尾追加验证记录：
  - 测试时间、测试环境（硬件型号 / 固件版本）
  - 通过/失败项、已知差异说明

**此阶段不修改任何代码**。若发现 bug，退回阶段 2 修复后重新验证。

**验收**：计划文件中有完整验证记录，所有测试项通过。

---

### 阶段 4 — 清除（Cutover）

**目标**：彻底移除老模块，清理所有迁移痕迹。

**操作**：
- 删除老模块全部源文件
- 删除 CMakeLists.txt / Kconfig 中的老模块引用和条件开关
- 将新模块变为唯一实现（去掉开关或设为默认）
- 更新 `docs/rules/plans/SUMMARY.md`，将该迁移任务状态改为 ✅ 已完成
- 确保编译通过、测试通过

**commit 格式**：`[REFACTOR] remove old <module>`

**禁止**：
- 不允许在此阶段引入新功能（只做删除和清理）
- 不允许保留"以防万一"的老代码注释块（用 git history 追溯）

---

## 阶段间强约束

| 约束 | 说明 |
| ---- | ---- |
| 每阶段独立提交 | 每个阶段对应独立 Git commit，不合并阶段 |
| 主线始终可构建 | 任何阶段结束时 `make cm7` 必须零错误通过 |
| 禁止大跨步 | 不允许跳过阶段（如直接从阶段 1 跳到阶段 4 删除老代码） |
| 验证先于清除 | 阶段 4 必须在阶段 3 验证记录完成后执行 |

---

## 适用范围

以下情况必须触发渐进迁移流程：

- 引入新框架替换现有框架（如替换消息总线）
- 重构公共接口（如修改驱动抽象层接口定义）
- 重构模块目录结构
- 替换底层驱动实现（如从轮询改为中断/DMA）
- 任何"改完之后旧路径不再存在"的变更

以下情况**不需要**渐进迁移（但仍需遵守规则 0 重大改动先写计划）：

- 纯 bugfix（不改接口，只改实现）
- 新增独立功能（没有旧模块需要替换）
- 文档、配置、测试的独立修改
